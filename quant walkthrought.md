## 1. 先修知识：什么是模型量化？

### 1.1 一张图看懂

```
FP16 模型（原始）
  每个权重占用 16 bit（2 字节）
  例如 Qwen3-4B 的 40 亿参数 → 约 8 GB 显存
        │
        ▼  量化（Quantization）
INT4 模型（量化后）
  每个权重占用 4 bit（0.5 字节）
  同样的 40 亿参数 → 约 2.5 GB 显存
  压缩比 ≈ 3 倍
```

### 1.2 量化是怎么"压缩"的？

想象你有一张照片，每个像素的颜色用 0 到 65535（16-bit）来记录。但其实人眼分辨不出这么精细的差别。你可以做一个映射：把所有颜色值除以 256，缩放到 0~255（8-bit）。照片文件变小了一半，但看起来几乎没区别。

模型量化也是这个思路：**把权重的数值精度从 FP16 降到 INT4，但尽量保持计算结果不变**。

```
原始权重（FP16）：  [0.234, -0.567, 1.890, -2.345, ...]
                         │
                         ▼  scale = max_abs / 7
量化后（INT4）：    [1, -3, 7, -7, ...]  加上 scale=0.27
                         │
                         ▼  推理时反量化
恢复（FP16 近似）：  [0.270, -0.810, 1.890, -1.890, ...]
```

这里 **scale（缩放因子）** 是关键：scale 选得好，恢复后的值就接近原始值；scale 选得不好，精度就损失严重。

### 1.3 为什么量化后推理更快、更省显存？

| 维度 | FP16 | INT4 量化 | 变化 |
|------|------|-----------|------|
| 模型文件大小 | ~7.6 GB | ~2.5 GB | **减 3 倍** |
| 推理时显存占用 | 加载全部 FP16 权重 | 加载 INT4 权重 + 反量化中间结果 | **减 2-3 倍** |
| 推理速度 | 基准 | 更快（权重搬运带宽减 3 倍） | **快 1.5-2 倍**（batch size=1） |

> **关键洞察**：小 batch 推理时（比如单条对话），GPU 大部分时间在搬运权重（内存带宽瓶颈），而不是做矩阵乘法（计算瓶颈）。权重变小 3 倍 → 搬运速度变快 3 倍 → 推理变快。但如果 batch 很大变成计算瓶颈，量化就没什么加速效果了（反而反量化有开销）。

## 2. AWQ 的思想：为什么是激活感知？

### 2.1 朴素量化的局限

最简单的量化方案是：**给每一行权重单独算一个 scale**（即 per-channel 量化）。这样每个通道的 scale 能保留各自的信息。

但这里有个问题：**并不是所有权重都同样重要**。

考虑一个具体的例子。假设某一行权重对应的输入激活值：

```
通道 0：权重 = [0.5, -0.3, 0.1, -0.8]    对应的输入激活值通常 ≈ 10.0
通道 1：权重 = [0.5, -0.3, 0.1, -0.8]    对应的输入激活值通常 ≈ 0.001
```

通道 0 的贡献是 `10.0 × weight`，通道 1 的贡献是 `0.001 × weight`。同样精度损失对通道 0 的影响是通道 1 的 10000 倍！朴素量化给两个通道分配相同的精度——这对通道 0 不公平。

### 2.2 AWQ 的核心直觉

**"我们应该保护那些与大的激活值相乘的权重通道。"**

AWQ（Activation-aware Weight Quantization）的做法是：

```
激活值大的通道 → 给更多精度（scale 小一点，量化更精细）
激活值小的通道 → 可以省精度（scale 大一点，量化更粗糙）
```

实现的数学技巧：**在量化前先缩放权重和激活值**

```
原始计算：  Y = W × X
缩放后：    Y = (W × s) × (X / s)    ← s 是 per-channel 的缩放因子
                       │       │
                       │       └── 前一层除以 s（把激活"拉小"）
                       └── 权重乘以 s（把权重"放大"）
```

为什么要这么做？因为权重大了之后，量化时相对误差就小了（和用更细的尺子量同一个道理）。而激活值除以 s 后变小了，不影响最终结果——两者刚好抵消。

这就是 AWQ 论文的核心：**找到一组最优的 per-channel 缩放因子 s，使得量化后的输出尽可能接近 FP16 原始输出。**

## 3. 本次量化实战：逐步拆解

我们在 `/root/AutoAWQ/quantize_qwen3.py` 中完成了对 Qwen3-4B 的量化。下面是每一步的解读。

### 3.1 量化配置一览

```python
quant_config = {
    "zero_point": True,       # 使用零点量化（非对称量化）
    "q_group_size": 128,      # 每 128 个权重共享一个 scale
    "w_bit": 4,               # 4-bit 量化
    "version": "GEMM",        # GEMM 后端（适合 batch≥1，大上下文）
}
```

**这些参数是什么意思？**

#### `w_bit: 4`
每个权重用 4 bit 存储。4 bit 能表示 2⁴ = 16 个不同的值。

#### `q_group_size: 128`
不是给整行一个 scale，而是把一行分成若干"组"，每 128 个权重共享一个 scale。

```
一行权重有 2560 个值：
  └── group_size=128 → 分成 2560/128 = 20 组
      每组有自己的 scale 和 zero_point
      组内精度更高，组间灵活适应不同分布
```

**类比**：相当于把一张大照片切成小块，每块用不同的对比度参数来压缩——比整张照片用同一个对比度效果更好。

#### `zero_point: True`
不使用中点对称（INT4 范围 -8~7），而是用完整的 0~15 范围。好处是：如果权重分布不对称（例如大量接近 0 的正值），零点量化能更好地保留信息。代价是计算稍微复杂（反量化时多了减法）。

#### `version: "GEMM"`
选择 GEMM 内核。两个选择：

| 版本 | 适用场景 |
|------|---------|
| GEMM | batch size ≥ 1，大上下文友好，通用 |
| GEMV | batch size = 1 时快 ~20%，但不适合长上下文 |

### 3.2 第一步：加载 FP16 模型

```python
model = AutoAWQForCausalLM.from_pretrained(model_path)
```

这做了三件事：
1. 从 HuggingFace / 本地加载标准的 FP16 模型
2. 读取 `config.json` 得知 `model_type = "qwen3"`
3. 通过 `AWQ_CAUSAL_LM_MODEL_MAP` 找到 `Qwen3AWQForCausalLM` 作为包装类

**框架如何选择正确的模型实现？**

```
config.model_type = "qwen3"
        │
        ▼
AWQ_CAUSAL_LM_MODEL_MAP["qwen3"] → Qwen3AWQForCausalLM
                                        │
                          ┌─────────────┼─────────────┐
                          │ 知道 layer 结构             │ 知道哪些层要一起缩放
                          │ Layer = Qwen3DecoderLayer  │ Q/K/V 投影一起缩放
                          │ 共 36 层                    │ gate/up 投影一起缩放
                          └───────────────────────────┘
```

### 3.3 第二步：准备校准数据

```python
calib_texts = [
    "The history of artificial intelligence began in antiquity...",
    "量子计算利用量子力学的基本原理...",
    # ... 共 30 条双语文本
]
```

**校准数据是做什么的？**

AWQ 需要知道"激活值大概是多少"才能算最优 scale。校准数据不需要和任务数据一模一样——只需让模型前向传播一遍，记录中间层的激活值分布。一般来说 **128 条、每条 512 token 就足够稳定**。

我们用了 30 条中英双语文本来节省时间（因为是演示），论文推荐 128 条。

### 3.4 第三步：逐层量化（核心！）

```python
model.quantize(tokenizer, quant_config=quant_config, calib_data=calib_texts)
```

这是整个过程的核心，我们逐层拆解。Qwen3-4B 有 36 个 Transformer 层，对每一层做以下 4 步：

#### 步骤 A：收集输入特征（`_get_input_feat`）

把校准数据喂给模型，用 hook（钩子函数）拦截每一层中每个 `nn.Linear` 层的输入：

```
校准文本 → Embedding → Layer 0 → Layer 1 → ... → Layer N
                              │
                         forward hook 拦截
                              │
                         input_feat["self_attn.q_proj"] = Tensor(128, 2560)
                         input_feat["self_attn.k_proj"] = Tensor(128, 2560)
                         input_feat["self_attn.v_proj"] = Tensor(128, 2560)
                         ...
```

这样我们知道了：对于每个线性层，它的输入激活值在真实数据下长什么样。

#### 步骤 B：搜索最优 Scale（`_search_best_scale`）

**这是 AWQ 的核心算法。** 对于每一组需要一起缩放的层（比如 Q/K/V 投影），做：

1. **计算 w_mean**（权重在每通道的平均相对大小）
2. **计算 x_mean**（输入激活值在每通道的平均大小）
3. **网格搜索最优 α**：对 20 个候选 α 值（0, 0.05, 0.10, ..., 0.95, 1.0），计算：
   ```
   scale = x_mean^α / w_mean^(1-α)
   ```
   然后将 `scale` 应用到权重上（W × scale），再做伪量化（模拟 INT4 精度），计算输出与 FP16 原始输出之间的 L2 误差。**选误差最小的 α。**

```
α=0:  scale 完全由权重决定 → 保护权重大的通道
α=0.5: 平衡权重和激活值的影响
α=1:  scale 完全由激活值决定 → 保护激活值大的通道
```

4. **保留最优 scale**，对 Q/K/V/gate/up/down 投影分别做。

> **为什么 Q/K/V 要一起缩放？** 因为它们共享同一个输入（来自 input_layernorm），如果各自独立缩放，会破坏它们的输入一致性。一起缩放意味着用同一套 scale 处理共享的 norm 层。

#### 步骤 C：应用 Scale（`apply_scale`）

找到最优 scale 后，实际应用到模型参数上：

```
input_layernorm.weight  /= scale       # 前一层除以 scale
self_attn.q_proj.weight *= scale       # Q 投影乘以 scale
self_attn.k_proj.weight *= scale       # K 投影乘以 scale
self_attn.v_proj.weight *= scale       # V 投影乘以 scale
```

这样确保了数学上等价（Y 不变），但权重的数值分布更"平坦"——更不容易在量化中损失信息。

#### 步骤 D：可选裁剪（`_search_best_clip`）

除了缩放，还可以**裁剪**权重的极端值：

```
原始权重范围：[-5.2, 6.8]  →  裁剪后：[-3.1, 3.1]
```

为什么有用？INT4 的范围有限（-8~7 或 0~15），少数极端值会"撑大" scale，让大部分普通值的表示精度变差。裁剪掉这些 outlier 反而整体更好。

> 注意：Q/K/V 层通常不做裁剪，因为注意力机制对权重的精细结构敏感。

#### 步骤 E：实际量化权重（`_apply_quant`）

最后一步：把缩放后的 FP16 权重真正量化成 INT4。

```
W_fp16 (2560, 2560) → 分组 (2560, 20组×128) → 每组算 scale/zero
                                                      │
                                                      ▼
                    qweight: packed INT4 (2560, 1280)  ← 两个 INT4 打包进一个 INT8
                    qzeros:  packed INT4 (2560, 20)
                    scales:  FP16 (2560, 20)
```

### 3.5 第四步：保存量化模型

```python
model.save_quantized(quant_path)
tokenizer.save_pretrained(quant_path)
```

保存的文件结构：


## 4. 原理深度解析：scale 是怎么算出来的？

### 4.1 公式推导

AWQ 的优化目标是：

```
L(s) = || Q(W × s) × (s⁻¹ × X) - W × X ||²
```

翻译成人话：**找到缩放因子 s，使得"权重缩放→量化→反缩放"后做的矩阵乘法，结果尽可能接近原始的 FP16 矩阵乘法。**

其中：
- `W`：原始 FP16 权重矩阵
- `X`：校准数据的输入激活值
- `s`：per-channel 的缩放因子
- `Q(·)`：量化函数（把 FP16 映射到 INT4 再映射回来）

### 4.2 搜索过程可视化

```
α = 0.00  →  scale 由权重大小主导  →  量化误差: 0.032
α = 0.05  →  ...                  →  量化误差: 0.028
α = 0.10  →  ...                  →  量化误差: 0.024
...
α = 0.45  →  ...                  →  量化误差: 0.018  ★ 最优
...
α = 1.00  →  scale 由激活值主导  →  量化误差: 0.041
```

网格搜索 20 个 α 值，选误差最小的那个——简单但有效。

### 4.3 Scale 如何应用到模型

具体到代码层面，有三类 scale 应用场景：

| 前置层类型 | 使用函数 | 操作 |
|-----------|---------|------|
| RMSNorm / LayerNorm | `scale_ln_fcs` | norm.weight /= s; linear.weight *= s |
| nn.Linear → nn.Linear | `scale_fc_fc` | fc1.weight 最后一截 /= s; fc2.weight *= s |
| GELU / 激活函数 | `scale_gelu_fc` | 在激活函数后插入缩放；fc.weight *= s |

每种操作都确保：**数学上 Y 不变，但权重的数值分布更好量化。**

## 6. 常见问题与进阶

### 6.1 为什么不能所有模型都无痛量化？

不同模型对量化的敏感度不同：
- **Decoder-only 模型（Llama、Qwen）**：通常量化友好，性能损失极小
- **MoE 模型（Mixtral）**：需要特殊处理，gate 层不能量化
- **视觉语言模型（LLaVA）**：vision encoder 部分通常保持 FP16
- **小模型（<1B）**：每个权重都更重要，量化损失相对更大

### 6.2 group_size 怎么选？

- **128**（推荐）：速度和精度平衡，大多数模型的最佳选择
- **64**：更精细，文件稍大，精度更高
- **-1**（per-channel）：最粗糙，文件最小，精度损失最大

### 6.3 export_compatible 是什么？

```python
model.quantize(tokenizer, quant_config=quant_config, export_compatible=True)
model.save_quantized(...)  # 保存带 scale 但未 pack 的权重（GGUF/其他格式兼容）
model.pack()               # 真正量化
model.save_quantized(...)  # 保存 INT4 权重（CUDA 推理兼容）
```

适用场景：你想导出到其他推理框架（llama.cpp、GGUF 等），同时也需要 AutoAWQ 的原生推理格式。

### 6.4 本次使用的关键脚本

- **量化脚本**：[quantize_qwen3.py](quantize_qwen3.py) — 你可以在里面修改 `model_path`、`quant_path`、`quant_config` 来量化其他模型
- **推理测试**：[inference_test.py](inference_test.py) — 加载量化模型、生成回复的测试脚本
- **基准测试**：`python examples/benchmark.py --model_path <quant_path> --batch_size 1`

### 6.5 一句话总结

> **AWQ 不是直接量化权重，而是先分析"哪些权重通道对结果重要"（通过看激活值大小），然后给重要通道更多精度，不重要通道少给精度。实现手段是对权重和激活值做可逆缩放（一个乘 s，一个除 s），使得量化误差集中在不重要的通道上。**
